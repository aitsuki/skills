# 开发分支收尾设计

## 目标

新增一个可重复调用的 `finishing-a-development-branch` skill，用于安全整理
执行实施计划时产生的任务级提交，并引导用户完成开发分支的后续处置。

设计文档和实施计划的提交保持不变。只有实施过程中产生的提交可以被整理。

## 架构

工作流由两个相互协作的 skill 组成：

- `executing-plans` 负责执行并验证计划。在开始实施前，将当前 `HEAD`
  记录为 `implementation-base`。实施成功后，将流程交接给
  `finishing-a-development-branch`，并传递该边界。
- `finishing-a-development-branch` 负责所有开发分支收尾行为，包括安全检查、
  可选的提交整理、结果验证，以及最终选择合并、创建 PR 或保留分支。

`finishing-a-development-branch` 也可以由用户手动调用。手动调用时没有可信的
实施边界，因此 Agent 必须推断并展示候选基准提交，在修改历史前获得用户明确确认。

## 实施提交边界

由 `executing-plans` 交接时，唯一允许整理的提交范围为：

```text
implementation-base..HEAD
```

`implementation-base` 在设计文档和实施计划完成提交后、正式实施前记录，因此这两个
文档提交天然位于可整理范围之外。

手动调用时，Agent 分析近期本地提交历史并提出候选基准。继续执行前必须展示：

- 候选基准提交；
- 将保持不变的提交；
- 可以被重写的提交。

用户必须明确确认或修正该边界。Agent 不得仅依据自身推断重写历史。

## 前置条件与安全门

任何历史重写开始前，必须依次完成以下检查：

1. 实施相关测试通过。
2. 工作区和暂存区均为干净状态。
3. 确认当前分支名称。`main`、`master` 等主分支名称本身不阻止重写，候选提交是否已经
   发布才是安全边界。
4. 成功刷新远端引用。
5. 可整理范围中的提交均不能被任何远端跟踪分支引用。
6. 向用户展示可整理的提交范围。
7. 向用户展示拟执行的重写方案并获得明确批准。
8. 创建名称唯一的备份分支，并验证它指向重写前的 `HEAD`。

如果无法刷新远端状态，或者无法证明相关提交尚未推送，skill 必须拒绝重写历史。

备份分支采用以下不易冲突的命名格式：

```text
backup/<current-branch>-before-finish-<YYYYMMDD-HHMMSS>
```

skill 永远不会删除备份分支。流程结束时应报告分支名称，并提供用户验证无误后可手动
执行的删除命令。

## 动态提交整理决策

skill 不提供固定数量的选项。完成范围检查后，Agent 必须先统计并分析实施提交，再决定
是否需要询问用户：

- 没有实施提交时，跳过提交整理。
- 只有一个实施提交时，说明无需整理并直接进入分支处置，不向用户展示无意义的提交
  整理选项。
- 有多个实施提交时，先分析每个提交的目的、文件、diff、测试与依赖关系，再判断是否
  存在有价值的多语义提交方案。

如果提交较少，或者除“全部合并为一个提交”外不存在更清晰的分组方式，只展示：

1. 保持现有提交。
2. 将全部实施提交合并为一个提交。

如果存在有价值的多语义提交方案，Agent 必须先完整展示该方案，再提供：

1. 保持现有提交。
2. 将全部实施提交合并为一个提交。
3. 按已展示的语义提交方案整理。

不得把“是否需要分析多语义方案”作为选项交给用户，也不得使用“由 Agent 规划后再
展示”这类尚未提供实际信息的选项。用户做选择时必须已经看到提交标题、文件、摘要及
必要的排序和依赖关系。

选择保持现有提交时，不创建备份分支，也不重写历史。

### 合并为一个提交

Agent 提议最终提交信息，并在请求批准前展示完整操作方案。获得批准并创建备份分支
后，在 `implementation-base` 之上将可整理范围重建为一个提交。

### 整理为多个语义提交

Agent 应根据可以独立解释的行为变更进行分组，不得按计划任务编号或文件类型机械
分组。测试必须与其验证的行为放在同一个提交中。每个拟议提交都应使仓库保持有效，
并尽可能处于测试通过状态。

在展示提交处理选项前，Agent 必须为每个拟议提交展示：

- 提交标题；
- 涉及的文件；
- 简明的变更摘要；
- 必要时说明排序和依赖关系。

用户选择该方案后可以批准或要求调整。在最终分组方案获得批准前，skill 不得重写
历史。

提交历史应从 `implementation-base` 开始，按照获批的分组方案重新构建。工作流不得
依赖未提前向用户展示结果的不透明交互式 rebase。

## 验证

重写前，skill 记录：

- 原始 `HEAD`；
- 备份分支；
- 可整理范围最终生成的 Git tree；
- 从 `implementation-base` 到原始 `HEAD` 的 diff。

重写后，skill 验证：

- 备份分支仍然指向原始 `HEAD`；
- 新 `HEAD` 与原始 `HEAD` 具有相同的最终 Git tree；
- 从 `implementation-base` 到新 `HEAD` 的 diff 与重写前记录的 diff 等价；
- 实施相关测试再次通过。

只有全部检查通过后，工作流才能进入分支处置阶段。

## 分支处置

完成提交处理和验证后，skill 提供从上游 `finishing-a-development-branch`
工作流调整而来的开发分支处置选项：

- 在本地合并该分支；
- 推送该分支并创建 PR；
- 保持该分支现状。

如果当前分支本身就是确认后的目标分支，不展示“本地合并”或“从同一分支创建 PR”等
无意义选项。此时说明变更已经位于目标分支，只提供推送或保持本地等适用操作。

删除性分支操作不属于提交整理。任何会删除提交、分支或 worktree 的选项都必须单独
获得明确确认，并且必须保留提交整理阶段创建的备份分支。

## 失败处理与恢复

创建备份分支前发生的失败不得修改提交历史。

创建备份分支后发生失败时，工作流必须立即停止。skill 不得自动回滚、删除引用、
继续执行分支处置，也不得自行改用另一种重写策略重试。

失败报告必须包含：

- 失败的操作；
- 当前分支和 `HEAD`；
- 备份分支及其指向的提交；
- 已完成的验证结果；
- 恢复命令，例如：

```bash
git reset --hard <backup-branch>
```

恢复命令仅供用户参考，skill 不得自动执行。

## 对现有 Skill 的修改

### `executing-plans`

- 在第一个实施任务开始前记录 `implementation-base`。
- 保留现有的计划执行和最终验证职责。
- 验证成功后调用 `finishing-a-development-branch`，不再直接结束。
- 将记录的实施边界传递给收尾工作流。
- 不自行执行提交历史重写。

### `writing-plans`

保留任务级提交步骤。它们在执行期间提供有用的检查点，只有用户在开发分支收尾阶段
选择整理时才会被合并。

### `brainstorming` 及计划文档提交

保留现有的设计文档和实施计划提交要求。这些提交位于 `implementation-base` 之前，
永远不会进入实施提交整理范围。

## Skill 定义验证

实现时必须验证 skill 文档已无歧义地规定以下行为：

- `executing-plans` 会记录并传递实施边界；
- 手动调用时，必须由用户确认 Agent 推断的边界；
- 单个实施提交会跳过提交整理询问；
- 多个实施提交会在询问前完成语义分析，并根据分析结果动态展示两个或三个选项；
- 第三个选项出现时，用户已经看到完整的语义提交方案；
- 工作区不干净、提交已发布或远端状态无法验证时，必须阻止重写；
- 位于 `main`、`master` 等主分支时，只要候选提交已被证明尚未发布且其他安全门通过，
  就允许重写；
- 两种提交整理模式都会创建并保留备份分支；
- 多语义提交方案必须在重写前展示并获得批准；
- 重写前后的 Git tree 和 diff 必须等价；
- 重写前后都必须运行测试；
- 任何路径都不会自动删除备份分支；
- 设计文档和实施计划提交不可能进入可整理范围。
